## Rules

1. **Tasks must be atomic** — each task should be completable in one coding session (30–90 min). If a task is too large, split it.
2. **Tasks must be ordered** — dependency order. A task that depends on another must come after it.
3. **Every task needs a validation command** — always `npm run build`, `python -m pytest`, or equivalent.
4. **implementation_notes must be concrete** — include exact file paths, component names, CSS classes, API routes, and type names. No vague guidance like "style it nicely" or "handle errors". Write as if the developer cannot see any other file.
4a. **Every file the task must create or modify MUST be listed with a backtick-quoted path in implementation_notes** — e.g. `` `src/app/users/page.tsx` ``. If the task spans multiple files (e.g. "refactor X across all callers"), list each call site as a separate backticked path. The downstream builder agent only edits files it sees explicit paths for — omitted paths will not be touched.
5. **UI tasks must specify visual details** — layout (flex/grid), spacing (Tailwind classes), colors (use the project's palette/tokens), component variants. If a Figma design is provided, reference it directly.
6. **acceptance_criteria must be visual and testable** — describe what the user sees, not just "it works". Include at least one build/test command criterion.
7. **Do not include infrastructure tasks** (CI/CD, Dockerfile, deploy scripts) unless explicitly mentioned in the requirements.
8. **Do not fabricate requirements** — only plan what is described in the README/PRD/Figma.
9. Output ONLY the three blocks. No preamble, no explanation, no markdown outside the blocks.
10. **Block 1 (json:issue_plan) must always be complete and valid JSON** — never truncate it.
11. **Be appropriately detailed** — task titles ≤ 10 words; descriptions 2-4 sentences; implementation_notes as many as needed (aim for 5-7 per task for UI work). The total JSON must fit within 28,000 tokens.
12. **Match the existing code style** — if package.json and existing files are provided, reference the exact libraries, hooks, and patterns already in use.
13. **Next.js App Router route groups**: a `(group)` directory's `layout.tsx` only wraps routes that **live inside that same directory**. Never tell the builder to create a primary file at `app/X/page.tsx` AND a `app/(group)/layout.tsx` and expect the group's layout to wrap X — it will not, and creating both will produce a duplicate-route build error if the builder also adds `app/(group)/X/page.tsx`. Choose ONE placement per route: either `app/(group)/X/page.tsx` (uses the group layout) OR `app/X/page.tsx` (no group layout). For any task that introduces a shared auth/marketing/dashboard layout, put the page inside the group from the start.
