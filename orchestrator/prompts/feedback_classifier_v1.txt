## Classification rules

Pick exactly one `kind`:

- **`issues`** — one or more concrete, well-scoped changes the user described. Each becomes its own GitHub issue. Use this for bug fixes, small UI tweaks, single-feature additions, **refactors that touch many files but have one goal** (e.g. "extract shared <DataTable> and use it everywhere" → ONE issue, not an epic), and anything where you can write specific acceptance criteria. If the user mentioned multiple genuinely distinct changes in one message, return multiple issues.

- **`epic`** — Reserve this for genuinely large, multi-feature scopes that introduce 2–5 independent surfaces. Good example: "Add an admin panel with user management, role assignment, and audit logs." Bad example: "Unify table styles across the app" — that's one refactor, not an epic. **If in doubt, prefer a single issue over an epic.** Return a parent issue (labels: ["epic"]) plus the sub-issues.

- **`clarify`** — the request is too vague to write acceptance criteria for ("make the dashboard better", "improve performance"). Ask a specific follow-up question. Return zero issues. Prefer this over guessing when scope is unclear.

For refactors that touch N callers, list each call site as its own backticked path. The builder only edits files for which it sees explicit paths — omit a path, and that file will not be touched.",

If `kind` is `clarify`, return an empty `issues` array.
If `kind` is `epic`, the first item in `issues` is the parent (label `epic`, broad description) and the rest are the sub-tasks.
